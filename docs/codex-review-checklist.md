# Codex 后续改动红线检查清单

适用范围：`临渊行` 官网当前 Stage 1A 及后续相邻阶段的代码审查。

使用目标：每次 Codex 改完后，优先判断有没有越界到“视觉大改”或“功能超范围”，其次再看结构是否更清晰、静态页是否仍可直接运行。

---

## 1. 先看范围是否越界

只要出现以下任一项，优先判定为高风险或直接不通过：

- 把纯静态站迁移到 React、Vue、Next.js、Vite 或其他框架。
- 引入后端、数据库、鉴权、登录态、后台管理、管理接口。
- 增加真实 API 请求、表单提交、预约收集、邮件发送、短信逻辑。
- 增加媒体上传、文件上传、管理发布、内容录入系统。
- 把当前首页重写成另一套结构或大规模重构 HTML / CSS / JS。
- 删除或破坏现有可用交互。

---

## 2. 视觉保护红线

### 2.1 首页视觉气质

检查是否保持以下内容基本不变：

- 首屏 Hero 的整体氛围是否仍是当前这版的国风、压暗、悬疑感。
- 页面整体配色是否仍以当前深色、金色点缀为主。
- 字体方向是否仍沿用当前中文 serif 气质。
- 模块顺序、视觉节奏、留白关系是否基本保持原样。
- 主标题、副标题、按钮、角色卡、地图卡、档案区是否仍维持原有视觉层级。

### 2.2 视觉越界信号

出现以下情况要重点拦：

- 大改背景图、整体配色、字体体系、主标题风格。
- 重做首屏布局，导致首页一眼看上去像另一版网站。
- 大改按钮、卡片、边框、阴影、分栏结构。
- 把原本克制的页面改得过亮、过花、过现代 SaaS 化。
- 新增区块明显抢走 Hero、PV、角色或地图模块的主视觉权重。

---

## 3. 功能边界红线

### 3.1 当前允许的内容

- 补结构注释、模块边界、锚点。
- 加静态导航入口。
- 加静态占位页。
- 加“敬请期待 / 预约开启”类纯文案占位。
- 补最基础 SEO。
- 写说明文档。

### 3.2 当前不允许的内容

- 真实登录表单、注册表单、验证码、找回密码。
- 真实预约表单、提交按钮联动、成功提示、数据收集。
- 新闻列表拉取、搜索、分页、后台发布。
- 下载按钮真实跳转、包体分发、平台检测、渠道逻辑。
- 用户中心、预约记录、活动资格查询。
- 接口封装、Mock API、数据模型提前工程化。

---

## 4. HTML 审查点

- 首页是否仍然可以直接打开 `index.html` 运行。
- 是否只是补充语义边界，而不是重写大段模块结构。
- 新增导航入口是否清楚标记为未来入口。
- 新增占位页是否明确写清“尚未开放 / 仅占位 / 未实现”。
- 是否保留现有模块 id、按钮属性、弹窗结构、视频结构。
- 是否误删现有交互依赖的 `data-*` 属性或 class 名。

---

## 5. CSS 审查点

- 是否只做增量样式，而不是重写大段旧样式。
- 是否复用现有颜色变量、字体方向、边框语言和阴影体系。
- 是否新增了会明显改变首页节奏的全局样式。
- 是否修改了 `.hero`、`.section`、`.site-nav`、`.character-*`、`.feature-*`、`.lightbox`、`.character-modal` 的核心布局。
- 是否因为新增导航导致桌面或平板首屏明显变形。
- 是否因为新增入口导致移动端反而看不到未来入口。

---

## 6. JavaScript 审查点

当前阶段原则：能不改 `script.js` 就不改。

如确实有改动，重点检查：

- 是否影响现有滚动按钮。
- 是否影响 PV 播放逻辑。
- 是否影响角色卡片视频、展开、弹窗。
- 是否影响地图灯箱切换。
- 是否影响 Lore 折叠展开。
- 是否新增登录、预约、下载、请求、存储等真实业务逻辑。

只要出现以下内容就要警惕：

- `fetch`
- `XMLHttpRequest`
- `localStorage` / `sessionStorage` 被用于账号或预约状态
- 表单提交监听
- 跳转到真实业务地址
- 新增大段配置化数据结构并推动整站数据化改造

---

## 7. 占位页审查点

新闻、预约、登录占位页应满足：

- 页面很简单。
- 复用现有视觉语气，但不比首页更抢眼。
- 明确说明“未来预留”。
- 不伪装成已完成系统。
- 不出现真实输入、提交、鉴权、拉取数据行为。

---

## 8. 文档审查点

- 是否说明当前模块边界。
- 是否说明未来预留区域。
- 是否明确写出当前禁止实现的内容。
- 文档是否帮助后续继续控范围，而不是引导大重构。

---

## 9. 每次审查的输出顺序

以后每次审查 Codex 改动时，按这个顺序给结论：

1. 先报“是否越界”。
2. 再报“明确发现的问题”，按严重度排序。
3. 再报“存在风险但未完全确认的问题”。
4. 最后报“总体是否可接受”。

---

## 10. 可直接复用的审查模板

### 审查结论

- 是否越界：通过 / 不通过
- 视觉风险：无 / 轻微 / 明显
- 功能越界：无 / 有
- 静态可运行性：通过 / 待验证

### Findings

- `[P1]` 是否出现功能越界。
- `[P2]` 是否出现视觉结构明显改动。
- `[P2]` 是否破坏现有交互。
- `[P3]` 是否新增了不必要的工程化铺垫。

### Residual Risk

- 是否存在需要人工肉眼确认的导航、首屏、移动端节奏变化。
- 是否存在新增占位页过重、过满、过像正式功能页的问题。

### Final Call

- 可接受，继续下一步。
- 需小修后再继续。
- 越界，建议回退或重做。
